# warehouse/ — the dbt project

This directory is the **transformation layer**: SQL models, run by
[dbt](https://docs.getdbt.com), that turn the raw tables the pipeline lands in
Postgres into the query-shaped marts the API serves. *Why* dbt (and not
hand-written SQL, or Spark) is recorded in
[ADR-0006](../docs/adr/0006-data-plane.md).

dbt is a *transformation* tool, not a database. It compiles your SQL models
into plain SQL, sends that SQL to Postgres to run, and Postgres does the
actual work. dbt itself stores no data.

## What lives here

You will see three kinds of things in this folder. Only the first is source
code — **the other three are generated or downloaded and must never be
edited or committed** (all three are in [`.gitignore`](../.gitignore)).

| Path | What it is | Tracked in git? |
|------|-----------|-----------------|
| `dbt_project.yml`, `models/`, `seeds/`, `tests/`, `profiles*` | **Source** — the SQL models and config you write | ✅ yes |
| `dbt_packages/` | **Downloaded dependencies** (see below) | ❌ no |
| `logs/` | **Run logs** — where dbt writes what it did | ❌ no |
| `target/` | **Compiled artifacts** — dbt's build output | ❌ no |

### `dbt_packages/` — just downloaded dependencies

The dbt equivalent of `node_modules/` (npm) or `.venv/site-packages/` (pip).
When `packages.yml` lists a package (e.g. `dbt_utils`, a library of reusable
SQL macros), running **`dbt deps`** downloads it into `dbt_packages/`. It is
third-party code we don't own, pinned by version, and reproducible from
`packages.yml` — so it is gitignored, exactly like `node_modules/`. Nothing in
here is hand-written.

*(ADR-0006 notes we aim to avoid `dbt_utils` unless a model genuinely needs a
macro — one fewer downloaded dependency to explain.)*

### `logs/` — where dbt run logs go

`logs/dbt.log` is dbt's verbose, append-only log of every invocation: the SQL
it compiled, the connections it opened, timings, and any errors. It is a
debugging aid, regenerated on every run, and grows without bound — so it is
gitignored. Delete it freely.

### `target/` — dbt compiled artifacts

dbt's build output: the compiled SQL, plus metadata files describing what was
built. **`target/` is a build
directory** (like `dist/` or `bin/obj`) — 100% regenerated by `dbt compile` /
`dbt run`, never edited, never committed.

## Running it

```bash
cd warehouse
dbt deps      # download packages listed in packages.yml -> dbt_packages/
dbt seed      # load CSV seeds (e.g. a sample route polyline) for CI/dev
dbt run       # compile models -> run the SQL in Postgres -> build marts
dbt test      # run the data tests (not_null / unique / accepted_values)
dbt build     # = seed + run + test in dependency order (the one to remember)
```

Connection details come from a dbt **profile** (host, db, user, password),
supplied via environment variables — never hard-coded — so the same models
target local Postgres and cloud Postgres by swapping the profile.
