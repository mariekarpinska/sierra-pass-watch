# warehouse/target/ — dbt compiled artifacts

**This is a build directory.** Everything in here except this README is
generated by dbt and is safe to delete — `dbt compile` / `dbt run` / `dbt build`
recreate it from scratch. Think `dist/` (frontend), `bin/`+`obj/` (.NET), or
`__pycache__/` (Python): output, not source. It is gitignored; only this
README is kept (via a negation in [`.gitignore`](../../.gitignore)) so the
explanation travels with the repo.

**Never edit anything here, and never commit the artifacts.** If a file here
looks wrong, fix the *model* in `warehouse/models/` and rebuild.

## What "compiled" means in dbt

You write models in `warehouse/models/` using SQL plus a little Jinja —
`{{ ref('stg_crashes') }}`, `{{ config(materialized='table') }}`, and similar.
Postgres cannot run Jinja. So dbt **compiles** each model: it resolves every
`ref()` / `source()` / macro into the real, fully-qualified table names and
plain SQL, and writes the result here. That compiled SQL is exactly what dbt
then sends to Postgres to execute. `target/` is where that intermediate and
final build output lands.

Two-step mental model:

1. **Compile** — Jinja + SQL model → plain SQL (written to `target/compiled/`).
2. **Run** — that SQL wrapped in `create table as …` / `create view as …`
   (written to `target/run/`) → sent to Postgres → the mart exists.

## What each file is

| Path | What it is | Why you'd look at it |
|------|-----------|----------------------|
| `compiled/sierra_corridor/…` | Your models with all Jinja resolved to **plain SQL** — no `create`, just the `SELECT` | Debugging: "what SQL did my macro actually produce?" Paste it into a client and run it as-is. |
| `run/sierra_corridor/…` | The compiled SQL **wrapped in the DDL** dbt executes (`create table … as select …`) | The exact statement dbt ran against Postgres. |
| `manifest.json` | The **full project graph** — every model, test, source, its columns, docs, and dependencies | The heart of dbt tooling: powers `dbt docs`, lineage, and state-based `--select state:modified` runs in CI. |
| `run_results.json` | **Results of the last invocation** — pass/fail, row counts, and timing per node | CI reads this to report what passed; you read it to find the slow model. |
| `graph.gpickle` | The dependency **DAG**, serialized (Python pickle) for dbt's internal use | You won't open it by hand; it's how dbt knows build order. |
| `graph_summary.json` | A lightweight summary of that graph | Tooling/debugging. |
| `semantic_manifest.json` | Metadata for dbt's **semantic layer** (metrics), if used | Ignore unless we adopt the semantic layer. |
| `partial_parse.msgpack` | A **cache** of the parsed project so the next run starts faster | Pure speed optimization. Delete it and dbt just re-parses. |

`compiled/` and `run/` mirror your `models/` folder structure under a folder
named after the dbt project (`sierra_corridor`).

## Why it's gitignored

Every file here is a deterministic function of your source models plus the
database state. Committing it would mean constant, meaningless diffs on every
run and merge conflicts in generated JSON — with zero information a reviewer
needs that the *source* models don't already carry. Standard practice: ignore
build output, commit the thing it's built from. (`manifest.json` is the one
artifact teams *sometimes* publish deliberately — as a CI cache to enable
"only rebuild what changed" — but that's an explicit artifact upload, not a
git commit.)
